# Reflection — introduction-to-builds

## What did I do?
For this challenge, I walked through the foundational build, run, and test loop for a Go application. After writing the initial server code in `main.go`, I compiled it using the standard `go build -o main main.go` command to produce a native executable. I executed the binary, which successfully started the server, and verified its functionality by querying port 4444 to ensure it returned the expected JSON payload. Next, I tackled the cross-compilation stretch task. I used the command `GOOS=linux GOARCH=arm64 go build -o main-arm64 main.go` to generate a binary targeted for an ARM architecture. To analyze the results, I ran the `file` command against both binaries to compare them. Finally, I intentionally attempted to execute the ARM binary on the Jenkins machine, which predictably resulted in an "Exec format error" since the host uses an x86_64 architecture.

## What was most surprising?
The most surprising detail was the difference in linking strategies revealed when I compared the outputs of the `file` command for both binaries. I expected the only difference to be the target architecture (x86-64 versus aarch64). However, the terminal explicitly showed that my native `./main` binary was dynamically linked, whereas the `./main-arm64` binary was statically linked. I was surprised to learn that standard Go builds default to relying on the host system's C libraries, resulting in a dynamically linked executable. Even more fascinating was realizing that cross-compiling automatically disables CGO. It does this assuming the host machine lacks ARM-specific compilers, forcing the linker to bundle pure-Go implementations and seamlessly producing a completely self-contained file without manual flags.

## What's still unclear?
While I learned a lot about build outputs, I still don't fully understand the Go linker and how it decides to use dynamic linking by default. Since I know cross-compiling turns CGO off, I am not entirely sure why a standard Go build needs to reach out to the operating system's C libraries (like glibc) in the first place. Because Go is its own language, I don't quite get which parts of my simple web server code actually require those external dependencies. I would like to learn more about how the linker makes these decisions behind the scenes and why it doesn't just statically link everything by default.
